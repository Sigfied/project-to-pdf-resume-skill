# 成果访谈与简历写作

本文件对应 `interview-and-writing.md`，用于追问项目成果，并把证据转成简历文案。

## 原则

- 先收集证据，再问成果。
- 问题要少，但要准。
- 指标、职责、采用情况和业务价值都视为需要用户确认的事实。
- 未经允许，不把机密细节写进最终表述。
- 为目标岗位写作，不是完整复述项目历史。

## 提问时机

成果问题必须在读完代码后提出，并基于已经发现的事实。

不好：

- “告诉我你的项目成果。”

更好：

- “仓库里能看到异步处理流程和状态跟踪。这个能力上线了吗？可以描述它带来的运维结果吗？”

## 问题集合

只问能增强 bullet 的最小问题集：

1. 哪些项目已经发布或被真实用户使用？
2. 谁在使用？
3. 有哪些规模或量级可以写？
4. 是否改善了性能、可靠性、成本、研发效率、用户体验或运营流程？
5. 这是迁移、替换、自动化、整合，还是新增能力？
6. 候选人的真实职责等级是什么？

如果用户已经给出宽泛但足够的信息，不要重复追问，把确认过的事实一致地用到文案里。

## 提问模板

可以这样问：

- `我看到 <technical capability> 的代码证据。它上线了吗？谁在用？`
- `这个工作能写规模吗，比如用户数、请求量、数据量、任务数、项目数或团队数？`
- `它的结果主要是可靠性、性能、成本、流程效率、产品能力，还是迁移替换？`
- `职责写 led、owned、core contributor、maintained、assisted 哪个更准确？`
- `有没有名称、数字、客户或内部系统是敏感的，需要泛化？`

不要让用户替你写简历。你只问事实，然后自己完成写作。

## Bullet 公式

使用这些结构：

- `负责/构建 + 模块或系统 + 技术方案 + 结果`
- `设计并实现 + 核心流程 + 关键机制 + 工程或业务收益`
- `搭建 + 平台/服务/工具 + 架构形态 + 支撑能力`
- `优化 + 问题领域 + 方法 + 量化或定性结果`

每条 bullet 检查四个部分：

| 部分 | 问题 |
| --- | --- |
| 动作 | 候选人做了什么？ |
| 对象 | 哪个模块、系统、流程或能力被改变？ |
| 方法 | 什么技术方案让这件事可信？ |
| 结果 | 对用户、系统、运营或维护者有什么变化？ |

中性示例：

- `构建内部平台的服务编排层，将 API 处理、业务流程和持久化逻辑分离，提升可维护性和发布信心。`
- `设计异步执行流程，覆盖请求提交、状态跟踪、worker 处理和结果获取，提高长耗时操作的可靠性。`
- `实现配置驱动的校验和元数据管理逻辑，减少人工配置，使新场景接入更可重复。`

## 动词强度

动词强度必须匹配证据：

| 动词 | 使用条件 |
| --- | --- |
| `主导` | 用户确认负责，或证据显示端到端职责。 |
| `负责` | 用户负责某模块、服务或项目区域。 |
| `设计并实现` | 用户确认设计工作，或代码清晰体现架构设计。 |
| `参与` | 职责不确定，或用户只是多名贡献者之一。 |
| `优化` | 存在前后变化、指标或清晰工程改进。 |

不要把每条都弱化成“参与”，也不要把每条都拔高成“主导”。

## 结论强度

把笔记转成简历文案时，用这个尺度：

| 结论类型 | 要求 |
| --- | --- |
| 技术实现 | 代码或文档能看到机制。 |
| 架构职责 | 代码结构加用户确认，或有清晰 ownership 证据。 |
| 生产使用 | 用户确认上线、使用者或运维使用。 |
| 量化结果 | 用户确认准确指标，并允许写出。 |
| 业务价值 | 用户确认这项工作为什么重要。 |

如果只是合理推断，在笔记中弱化或继续追问，不要当成事实写进最终简历。

## 技术栈写法

避免原始依赖列表。

不好：

- `Tech stack: language, framework, database, cache, container runtime.`

更好：

- `框架负责 API 路由和服务组合，数据库存储领域记录，缓存支持快速查询或协调，容器配置保证可重复部署。`

## 没有指标时的成果

没有确认数字时，可以写可信的定性结果：

- 提升可维护性
- 减少人工操作
- 增强运维可见性
- 改善失败恢复
- 标准化重复流程
- 支撑明确用户群体的持续使用

不要编造百分比、数据量、采用范围或成本节省。

## 按岗位调整侧重点

| 目标 | 优先强调 |
| --- | --- |
| 后端 | API、服务边界、数据一致性、异步处理、可靠性、性能、部署。 |
| 数据 | 链路、建模、质量、调度、存储、查询效率、治理。 |
| 全栈 | 用户流程、前后端集成、产品完整度、可用性、交付。 |
| 平台 | 可复用能力、配置化、多租户或多项目支持、可观测性。 |
| AI/工具 | 自动化流程、工具集成、评估、人审闭环、安全边界。 |

同一份证据可以按不同岗位重写，不要原样复制 bullet。

## 修改前后示例

弱：

- `Worked on backend development using several technologies.`

更好：

- `构建服务流程的请求编排层，拆分校验、执行、持久化和状态回传，使长耗时操作可以被可靠管理。`

弱：

- `Used a database and cache to implement project features.`

更好：

- `围绕配置驱动流程实现持久化和缓存能力，减少重复配置工作，让新场景更容易接入。`

## 最终简历风格

- 匹配用户要求的语言和地区。
- bullet 要短，能放进最终版面。
- 最强、最相关的项目靠前。
- 主要项目通常保留 3-5 条 bullet，弱项合并。
- 只有技术能解释难度或价值时，才写具体技术。
- 不要每条重复同一串技术栈。
- 时态和语气保持一致。
- PDF 拥挤时，优先删背景解释，不要删有证据的成果。

## 红旗

这些 bullet 要修改或删除：

- 只列技术，不说明怎么用
- 没有来源就声称规模
- 未确认职责却写“主导”
- 内部业务细节写得太具体
- 多个项目重复相同方法和结果
- 像项目 README，不像候选人影响
- 需要读者自行推断为什么重要

## Review 清单

进入 PDF 前确认：

- 每条 bullet 都有动作、方法和结果。
- 指标已确认，或已删除。
- 技术结论有文件或用户确认支撑。
- 首页前半部分能看出目标岗位。
- 没有 bullet 只是技术栈列表。
- 没有缺乏证据的资历宣称。
